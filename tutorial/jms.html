Java Messaging Service - JMS


This tutorial is designed to provide information about JMS and provides a step-by-step procedure on how to implement JMS in your project.
Prerequisite for this tutorial would be to have basic knowledge on object-oriented paradigms and Java.

1. Introduction

JMS stands for Java Message Service which is a standard to access messages in messaging middleware. JMS provides an API to create, send and read messages on Java servers.

1.1 Messaging

It is a process of exchanging information between two or more applications. Applications can communicate with each other by passing or exchanging information (known as message). Messages are asynchronous requests or responses that will be generated by an application and may be consumed by other applications. "Messaging enables distributed communication that is loosely coupled. A component sends a message to a destination, and the recipient can retrieve the message from the destination. However, the sender and the receiver do not have to be available at the same time in order to communicate. In fact, the sender does not need to know anything about the receiver; nor does the receiver need to know anything about the sender. The sender and the receiver need to know only which message format and which destination to use."(Oracle and/or its affiliates., 2013)

1.2 Messaging Domains

There are two types of messaging model that JMS supports:

  • Point to point (PTP) – In this model the messages are created by an application and pushed into the Queue, the queue holds the messages until consumed by another application. It "allows a client to send a message to another client via an intermediate abstraction called a queue. The client that sends the message sends it to a specific queue. The client that receives the message extracts it from that queue" (Deakin, Nigel., 2013).

  • Publish/Subscribe (Pub-Sub) – In this model, the message is created by an application and published to a Topic, the same message is consumed by many other applications whenever required. It "allows a client to send a message to multiple clients via an intermediate abstraction called a topic. The client that sends the message publishes it to a specific topic. The message is then delivered to all the clients that are subscribed to that topic" (Deakin, Nigel., 2013).
